# General Rules

## 破壊的操作の制限

復元不可能な操作を防ぐため、以下の代替コマンドを使用する。

| 禁止コマンド | 代替コマンド | 理由 |
|---|---|---|
| `rm` | `trash` | ゴミ箱に移動し、誤削除からの復元を可能にする |
| `git checkout` | `git switch`（ブランチ切替）/ `git stash`（変更退避） | 変更の暗黙的な破棄を防ぐ |
| `git restore` | `git stash` | 変更を保全した上で復元する |
| `git clean` | `git stash -u` | 未追跡ファイルも含めて退避する |
| `git reset --hard` | `git reset`（`--hard` なし）+ `git stash` | `--hard` なしならワーキングツリーを破壊しない |

## `grill-me` スキル発火時の開発ワークフロー

`grill-me` スキルが発火された場合、以下のフローに沿って進めること。

| # | フェーズ | ツール | アクション |
|---|---|---|---|
| 1 | 設計詰め＆リサーチ | `grill-me` スキル / context7 MCP（フォールバック: WebSearch / WebFetch） | インタビューで設計を詰めつつ、公式ドキュメント・参考実装・論文を収集 |
| 2 | プラン作成 | plan mode | プランファイルを書き起こす |
| 3 | テスト駆動実装 | `tdd` スキル | プランファイルが承認された事を確認し、RED-GREEN-REFACTOR ループを回す |
| 4 | リファクタリング | `simplify` スキル | 再利用・品質・効率性を改善 |
| 5 | セキュリティレビュー | `security-review` スキル | ブランチ差分をチェック |

## 理由の提示

コードや設定を変更する際は、結果だけでなく根拠を示す。
ユーザーが判断・学習できるよう、以下の観点で簡潔に説明すること。

- 実装方針を選択した理由（例: 「X ではなく Y を使う理由は〜」）
- トレードオフがある場合はその比較
- 公式ドキュメントやベストプラクティスに基づく場合はその旨を明記

ただし、以下のような自明な変更には理由の説明は不要：

- typo 修正、フォーマット整形
- ユーザーが具体的な変更内容を指示した場合の単純な適用
- 変数名のリネームなど意図が明白な変更
